How Court Meta fits into your site

Court Meta is a Chrome extension and a local Windows service that together expose Indian eCourts data to any web page the user opens. Your site never talks to eCourts directly — you talk to a tiny JavaScript bridge that hands requests off to the extension, which talks to the local service, which talks to eCourts.

your-page.html
   │
   │   includes <script src="court-meta.js">
   │
   ▼
CourtMeta.fetchStates()      ←  promise-based JS API
   │
   │   window.postMessage
   ▼
content.js  ──>  background.js  ──>  http://localhost:5000  ──>  eCourts

What this means for you as an integrator:

  • Your site can be hosted anywhere — the bridge works regardless of origin, because it’s the user’s extension making the network calls, not your page.
  • You don’t need API keys, CORS configuration, or a backend proxy.
  • If the user doesn’t have the extension installed, every call rejects — you’re responsible for showing an install prompt. See Detecting the extension.

Setup

Two pieces — one for the user (install once), one for you (drop in a script).

1. The user installs Court Meta

Hand them court-meta-extension.zip with these instructions:

  1. Unzip to a folder you won’t move (Chrome remembers the path).
  2. Open chrome://extensions and turn on Developer mode.
  3. Click Load unpacked and select the unzipped folder.
  4. If you also installed the Windows installer, the local Court Meta API service is already running. If not, the extension will tell them.

2. Include the bridge in your page

<!-- somewhere before your app code -->
<script src="https://localhost:5000/court-meta.js"></script>

Or copy court-meta.js to your own server and serve it from there — the file is self-contained, has no dependencies, and only uses window.postMessage.

It’s a single global. Including the script defines window.CourtMeta. Every method returns a Promise.

Detecting the extension on page load

Always call CourtMeta.ready() first. It resolves when the extension answers the handshake (usually within ~50 ms) or rejects after a timeout if the extension is missing.

CourtMeta.ready()
  .then(() => {
    // extension is installed and the local service is reachable
    enableSearchUi();
  })
  .catch(() => {
    // not installed — show an install banner
    showInstallBanner();
  });

Pass a custom timeout if you want to wait longer than the default 2 000 ms:

CourtMeta.ready(5000).then(...);
You don’t need to call ready() before every method. Each API call internally awaits readiness and rejects with the same “extension not detected” error if it never arrives. But calling ready() upfront lets you swap UI states (install banner vs. real form) before the user clicks anything.

Workflow A · Lookup a case by CNRIf you have the CNR

A CNR is the 16-character national identifier for an Indian court case (e.g. MHPU010001232022). It uniquely resolves to one case, so this is the fastest path when your user already has one.

Use cnrBundle for new code

const cino = 'MHPU010001232022';

CourtMeta.cnrBundle(cino)
  .then(bundle => {
    // bundle has: caseInfo, parties, advocates, court,
    // filing, hearings, transfers, processes, orders
    console.log(bundle.caseInfo.title);
    bundle.hearings.forEach(h => console.log(h.date, h.purpose));
  })
  .catch(err => alert(err.message));

Form pattern

cnrInput.addEventListener('keydown', e => {
  if (e.key !== 'Enter') return;
  const cino = cnrInput.value.trim().toUpperCase();
  if (cino.length < 16) return showError('CNR must be at least 16 characters');

  setLoading(true);
  CourtMeta.cnrBundle(cino)
    .then(renderCase)
    .catch(err => showError(err.message))
    .finally(() => setLoading(false));
});
cnrSearch still exists and returns the legacy raw history blob alongside a parsed view. Prefer cnrBundle — it returns the structured shape directly with no envelope to unwrap.

Workflow B · Browse the court hierarchyCascading dropdowns

Use this when your user doesn’t know a specific case but wants to drill into a state, district, or court room. Every other search workflow needs a complex picked from this hierarchy, so this pattern is reused everywhere.

The four levels

States       => CourtMeta.fetchStates()
Districts    => CourtMeta.fetchDistricts(state_code)
Complexes    => CourtMeta.fetchComplexes(state_code, dist_code)
Court rooms  => CourtMeta.fetchCourts(state_code, dist_code, court_code)

Wiring three cascading <select>s

CourtMeta.fetchStates().then(({ states }) => {
  fillSelect(stateEl, states, s => [s.state_code, s.state_name]);
});

stateEl.addEventListener('change', () => {
  resetSelect(districtEl);
  resetSelect(complexEl);
  if (!stateEl.value) return;

  CourtMeta.fetchDistricts(stateEl.value).then(({ districts }) => {
    fillSelect(districtEl, districts, d => [d.dist_code, d.dist_name]);
  });
});

districtEl.addEventListener('change', () => {
  resetSelect(complexEl);
  if (!districtEl.value) return;

  CourtMeta.fetchComplexes(stateEl.value, districtEl.value).then(({ complexes }) => {
    fillSelect(complexEl, complexes, c => [c.njdg_est_code, c.court_complex_name]);
  });
});
Use njdg_est_code, not complex_code. Every downstream workflow (case search, filing search, cause list, advocate, court rooms) takes the establishment code, which is the njdg_est_code field.

Workflow C · Search by case numberType + number + year

Use this when the user has registration details (e.g. “C.S. No. 123 of 2024”) but no CNR. The result is every case in the selected complex that matches.

1. Load case types for the chosen complex

Case types vary per complex — load them lazily once a complex is selected:

CourtMeta.fetchCaseTypes({
  state_code, dist_code, court_code // court_code = njdg_est_code
}).then(({ caseTypes }) => {
  caseTypes.forEach(t => addOption(caseTypeEl, t.id, t.name));
});

2. Run the search

CourtMeta.searchByCaseNumber({
  state_code, dist_code,
  courts: court_code,         // or 'CODE1,CODE2' to search multiple
  case_type, number, year
}).then(({ count, results }) => {
  console.log(`${count} cases`);
  results.forEach(r => render(r.cino, r.case_no, r.petnameadArr));
});
Case-type IDs aren’t portable. The same label (“C.S.”) maps to different IDs in different complexes — always re-fetch when the complex changes.

Workflow D · Search by filing numberPre-registration

A filing number is what a case is given when it’s first lodged but before it’s registered. Use this when the matter hasn’t been numbered yet, or you only have the filing receipt.

CourtMeta.searchByFilingNumber({
  state_code, dist_code,
  courts: court_code,
  filingNumber, year
}).then(({ count, results }) => {
  results.forEach(r => render(r.cino, r.case_no, r.petnameadArr));
});

The response shape is identical to searchByCaseNumber — reuse the same row-renderer.

Workflow E · Cause list for a court roomDaily docket

A cause list is the schedule of cases listed before a particular court room on a particular date. Use this to render a docket view for a judge’s today (or any day).

CourtMeta.causeList({
  state_code, dist_code, court_code,    // the complex (njdg_est_code)
  court_no,                              // from fetchCourts(...)
  date: '15-04-2026',                    // dd-mm-yyyy or yyyy-mm-dd
  flag: 'civ_t'                          // 'civ_t' = civil, 'cri_t' = criminal
}).then(({ count, rows }) => {
  rows.filter(r => !r.isHeader).forEach(r => {
    if (r.cino) {
      // row carries a CNR — open the bundle directly
      renderRow(r.cino, r.cells);
    } else {
      // row only has caseNo + courtCode — resolve to a CNR first
      renderUnresolvedRow(r.caseNo, r.courtCode, r.cells);
    }
  });
});

Resolving a cause-list row to a CNR

Some cause-list rows don’t carry a CNR — they only carry the upstream’s internal case-number plus an establishment code. To open the bundle, resolve them first:

CourtMeta.cnrByCaseNo({
  state_code, dist_code,
  court_code: row.courtCode,    // note: row's own court_code, not the complex's
  case_no: row.caseNo
}).then(({ cino, data }) => {
  // data is already the parsed CaseBundle — no need to call cnrBundle again
  renderCase(data);
});

Workflow F · Search by advocateSingle or state-wide

One method, two scopes. The same call shape switches between a single round-trip and a streaming fan-out depending on scope.

Single court complex (synchronous)

CourtMeta.searchByAdvocate({
  scope: 'court',
  state_code, dist_code,
  courts: court_code,                // or 'CODE1,CODE2,...'
  mode: 'name',                      // or 'barcode'
  advocateName: 'John Doe',
  pendingDisposed: 'Pending'         // 'Pending' | 'Disposed' | 'Both'
}).then(({ count, results }) => {
  results.forEach(r => render(r.cino, r.petnameadArr));
});

Bar-code variant:

CourtMeta.searchByAdvocate({
  scope: 'court',
  state_code, dist_code, courts,
  mode: 'barcode',
  barstatecode: '1',
  barcode: 'MAH/12345/2010'
});

State-wide (live stream)

For state-wide search, the extension fans out to every district and streams results back. You supply an onProgress callback that fires as each district completes:

const aggregated = [];
const seen = new Set();

CourtMeta.searchByAdvocate({
  scope: 'state',
  state_code: 'MH',
  mode: 'name',
  advocateName: 'John Doe',
  pendingDisposed: 'Pending',

  onProgress(event) {
    if (event.type === 'start') {
      showStatus(`Searching ${event.totalDistricts} districts...`);
    }
    if (event.type === 'progress') {
      // dedupe across districts — same case can appear twice
      for (const row of (event.rows || [])) {
        if (row.cino && !seen.has(row.cino)) {
          seen.add(row.cino);
          aggregated.push(row);
        }
      }
      showStatus(
        `${event.completedDistricts} / ${event.totalDistricts} districts ` +
        `— ${aggregated.length} cases so far`
      );
      renderResults(aggregated);
    }
    if (event.type === 'complete') {
      showStatus(`Done — ${aggregated.length} cases.`);
    }
  }
}).then(({ progressCount }) => {
  // resolves once the trailing 'done' event fires
  console.log('stream finished after', progressCount, 'progress events');
}).catch(err => {
  // fatal stream errors land here; per-district errors come through
  // onProgress as event.error and are non-fatal
  showError(err.message);
});
State-wide search hits dozens of districts and can take minutes. Render results incrementally — don’t hide them behind a spinner.

Workflow G · Drill-downs from a CaseBundleHearings, orders, applications

Once you have a CaseBundle, every nested row points at a deeper API. These four helpers cover the drill-downs you’ll actually wire into UI.

Business notes for a hearing

Click on a row in bundle.hearings to see what the court actually recorded that day:

CourtMeta.cnrBusiness({
  state_code, dist_code, court_code,
  case_number: bundle.caseInfo.caseNumber,
  hearingDate: hearing.date,           // dd-mm-yyyy
  disposalFlag: hearing.disposalFlag,
  courtNo: hearing.courtNo
}).then(business => renderBusinessNote(business));

Application / writ detail

CourtMeta.cnrWrit({
  state_code, dist_code, court_code,
  case_number,
  app_cs_no: application.appCsNo
}).then(detail => renderApplicationDetail(detail));

Order PDF

Resolves to a data URL you can drop straight into an <iframe> or download anchor — no extra fetch needed:

CourtMeta.cnrOrderPdf({
  state_code, dist_code, court_code,
  orderYr: order.year,
  order_id: order.id,
  crno: order.crno
}).then(({ dataUrl, mime, size }) => {
  document.querySelector('iframe.preview').src = dataUrl;
  // or, for a download:
  const a = document.createElement('a');
  a.href = dataUrl;
  a.download = `order-${order.id}.pdf`;
  a.click();
});

Error handling

Every method rejects with an Error — the message is safe to display to users.

The errors you’ll actually see

MessageWhat it meansWhat to do
Court Meta extension not detected.… Extension not installed, or page loaded before the content script did. Show an install banner with the unzip + load-unpacked steps.
Court Meta extension is out of date… Old extension build is still installed (Chrome doesn’t auto-replace same-version unpacked builds). Tell the user to open chrome://extensions, click reload on Court Meta, then refresh the page.
Request timed out (action: …) 30 s passed with no response. Usually the local Windows service is stopped. Tell the user to open services.msc and start Court Meta API.
Upstream eCourts errors Anything else is propagated from eCourts (e.g. invalid CNR, no results). Show the message inline next to the form field.

Recommended pattern

function callApi(promise, { onSuccess, onError }) {
  return promise
    .then(onSuccess)
    .catch(err => {
      const msg = err.message || 'Unknown error';
      if (msg.includes('not detected')) {
        showInstallBanner();
      } else if (msg.includes('out of date')) {
        showReloadInstructions();
      } else {
        onError(msg);
      }
    });
}

API reference

Every method is on the global CourtMeta object and returns a Promise.

MethodResolves with
ready(timeoutMs?) void — rejects if the extension doesn’t answer within the timeout (default 2 000 ms).
fetchStates() { states: [{ state_code, state_name }, ...] }
fetchDistricts(state_code) { districts: [{ dist_code, dist_name }, ...] }
fetchComplexes(state_code, dist_code) { complexes: [{ njdg_est_code, court_complex_name, ... }, ...] }
fetchCourts(state_code, dist_code, court_code) { courts: [{ code, name, isHeader }, ...] } — court_code here is the njdg_est_code.
cnrSearch(cino) Legacy. { parsed, raw }.
cnrBundle(cino) Structured CaseBundle: caseInfo, parties, advocates, court, filing, hearings, transfers, processes, orders.
cnrByCaseNo({ state_code, dist_code, court_code, case_no }) { cino, data } — data is the same shape as cnrBundle.
cnrBusiness({ ..., case_number, hearingDate, disposalFlag, courtNo }) Business note for one hearing.
cnrWrit({ ..., case_number, app_cs_no }) Application / writ detail.
cnrOrderPdf({ ..., orderYr, order_id, crno }) { dataUrl, mime, size } — the dataUrl drops straight into an iframe.src or anchor href.
fetchCaseTypes({ state_code, dist_code, court_code }) { caseTypes: [{ id, name }, ...] } — cached server-side.
searchByCaseNumber({ state_code, dist_code, courts, case_type, number, year }) { count, results: [{ cino, case_no, type_name, reg_year, petnameadArr, establishment_name, ... }, ...], raw }
searchByFilingNumber({ state_code, dist_code, courts, filingNumber, year }) Same shape as searchByCaseNumber.
causeList({ state_code, dist_code, court_code, court_no, date, flag, selprevdays }) { count, html, rows: [{ index, isHeader, cino?, caseNo?, courtCode?, cells: [...] }, ...] }
searchByAdvocate({ scope: 'court' | 'state', mode, ..., onProgress? }) scope: 'court' → { count, results, raw }. scope: 'state' → { progressCount } after final done event; rows arrive via onProgress.
Need to inspect a request shape live? Open DevTools on a page with court-meta.js loaded, run a method, and the extension logs the round-trip to its own service-worker console (chrome://extensions → Court Meta → Inspect views: service worker).
